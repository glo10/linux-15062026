## Synthèse commandes

|                                          | Commande                         |
| ---------------------------------------- | -------------------------------- |
| Lister les disques                       | `lsblk`                          |
| Créer une partition                      | `sudo fdisk /dev/sdb`            |
| Formater en ext4                         | `sudo mkfs.ext4 /dev/sdb1`       |
| Créer un point de montage                | `sudo mkdir -p /mnt/data`        |
| Monter une partition                     | `sudo mount /dev/sdb1 /mnt/data` |
| Démonter une partition                   | `sudo umount /dev/sdb1`          |
| Afficher les systèmes de fichiers montés | `df -h`                          |
| Modifier les montages automatiques       | `sudo nano /etc/fstab`           |
| Tester `/etc/fstab`                      | `sudo mount -a`                  |
| Modifier la configuration GRUB           | `sudo nano /etc/default/grub`    |
| Régénérer la configuration GRUB          | `sudo update-grub`               |
| Remonter `/` en lecture-écriture         | `mount -o remount,rw /`          |

## Partie I

1. Extinction de la machine
```bash
sudo shutdown
```
2. Ajout du sécond disque
- Sélection la VM.
- Configuration → Stockage.
- Ajout nouveau disque dur virtuel.
    - Type : VDI (VirtualBox Disk Image).
    - Taille : 5 Go.
    - Allocation : Dynamique.
- Validation et enregistrement de la configuration.
-  [Article installation de Fedora](https://doc.fedora-fr.org/wiki/Netinstall_de_Fedora)
4. Liste des disques
```bash
lsblk
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda      8:0    0   10G  0 disk
├─sda1   8:1    0  9.5G  0 part /
└─sda2   8:2    0  512M  0 part [SWAP]
sdb      8:16   0    5G  0 disk
```
- Normalement */dev/sdb* mais le nom peut varier selon la config du matériel
5. Création de la partition avec gdisk
```bash
sudo gdisk /dev/sdb
# En mode interactif
#   1. Tapez ? pour voir la liste des commandes disponibles
#   2. Tapez n pour créer une nouvelle partition + Entrée
#   3. La partition commence à 1 (on laisse la valeur par défaut en appuyant sur Entrée)
#   4. Elle se termine à la fin, on prend tout l'espace donc Entrée pour la valeur par défaut
#   5. Tapez p pour vérifier que tout est ok
#   6. Tapez w pour enregistrer
```
6. Formatage de la nouvelle partition
```bash
sudo mkfs.ext4 /dev/sdb1 # ou mkfs -t ext4 /dev/sdb1
```
7. Point de montage de la partition dans le dossier /mnt/data
```bash
# Création du dossier avant le montage
mkdir -p /mnt/data
sudo mount -t ext4 /dev/sb1 /mnt/data
```
8. Vérifiez que l'espace est disponible avec la commande suivante
```bash
mount | grep sdb1
# ou bien
df -h
Filesystem      Size  Used Avail Use% Mounted on
/dev/sdb1       4.8G   24M  4.5G   1% /mnt/data
```
- On devrait voir la partition /dev/sd1 avec 1Go d'espace disponible montée sur /mnt/data

9. Nouveau dossier
```bash
mkdir -p /mnt/data/test
```

## Partie II

10. reboot
```bash
sudo reboot
```
11. Après le redémarrage de la machine, résultats et conclusions des commandes suivantes
```bash
df -h
ls -lah /mnt/data
```
- La partition /dev/sdb1 n'existe plus
- Le dossier /mnt/data existe toujours mais il est vide
- On en déduit que la partition n'est pas montée au démarrage, le montage précédent était temporaire, après redémarrage elle n'est pas montée automatiquement donc il faut tout refaire et éditer le fichier /etc/fstab pour effectuer le montage au démarrage.
12. Modification du fichier /etc/fstab
```bash
sudo nano /etc/fstab
# contenu à ajouter
/dev/sdb1   /mnt/data   ext4    defaults    0   2
# Le contenu correspondant respectivement à la partition (périphérique à monter), dossier de montage (point de montage), type de fichier (sytème de fichiers), paramètres par défaut (options de montage, on verra plus tard ces options, par exemple les quotas), 0 et 2
```
- Tableau recap

| Champ       | Signification                                                              |
| ----------- | -------------------------------------------------------------------------- |
| `/dev/sdb1` | Partition à monter                                                         |
| `/mnt/data` | Point de montage                                                           |
| `ext4`      | Type de système de fichiers                                                |
| `defaults`  | Options de montage par défaut                                              |
| `0`         | Pas de sauvegarde avec `dump`                                              |
| `2`         | Vérification du système de fichiers après la partition racine au démarrage |
- Fortement recommandé d'utiliser UUID à la place du nom du disque pour éviter les problèmes si l'ordre des disques change dans le futur.
13. Test et montage
```bash
fstab
df -h /dev/sdb1 # la partition n'est pas montée
sudo mount -a # la partition est montée
```
- mount -a va monter tous les systèmes de fichiers déclarés dans /etc/fstab qui ne sont pas encore montés
- On devrait voir la partition avec sa taille, l'espace utilisé, l'espace disponible et le point de montage
14. Redémarrage
```bash
sudo reboot
```
15. Après redémarrage
```bash
df -h
ls -lah /mnt/data
```
- /dev/sdb1 est dans le résultat de la commande df -h
- /mnt/data/test est visible
16. Après démontage, le dossier /mnt/data/test est à nouveau présent
```bash
sudo umount /dev/sdb1
# vérification
df -h
ls -lah /mnt/data
```
- /dev/sdb1 n'apparaît plus
- Le dossier /mnt/data est présent
- Le dossier /mnt/data/test a disparu, les données ne sont pas supprimées mais inaccessibles car la partition n'est plus monté
17. 
```bash
sudo mount /mnt/data
# vérification
ls -lah /mnt/data
```
- /mnt/data/test réapparaît
- Conclusion générale : 
    - montage rend accessible le système de fichiers dans l'arborescence des dossiers Linux
    - démontage ne supprime pas les données mais elles sont temporairement inacessibles

## Partie III

18. Edition fichier config de GRUB
```bash
sudo nano /etc/default/grub
# Modification de cette ligne
GRUB_TIMEOUT=2
# sauvegarde MAJ
```
19. En déduit que la machine va démarrer plus rapidemment car la variable GRUB_TIMEOUT définit le nombre de sécondes pendant lesquelles l'utilisateur peut choisir son mode de démarrage depuis le menu GRUB, après ce délai, le système démarre par l'entrée par d"faut (l'autre paramètre GRUB_DEFAULT que vous avez vu lors de la modification de la config). Autrement dit ici, le système va potentiellement démarré plus vite que précédemment
20. Appliquez ses changements au prochain démarrage
```bash
# Debian
sudo update-grub
# Fedora
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
```

## Bonus

21. Modifiez le fichier /etc/fstab pour introduire une erreur
```bash
# dossier /mnt/error n'existe pas
/dev/sdb1   /mnt/error   ext4    defaults    0   2
```
22. Redémarrez la machine, normalement vous serez basculé en mode "Emergency Mode"
```bash
sudo reboot
```
23. Trouvez le moyen de corriger votre erreur dans le fichier /etc/fstab à l'aide de la commande mount
```bash
# En mode rescue, les fichiers sont en lecture seule et peu de services sont démarrés
# D'abord regarder le journal pour voir les éventuelles erreurs
journalctl --system -k # ou journalctl -a
# Passer en mode écriture
mount -o remount,rw /
# Ici on connait l'erreur et on le voit dans le journal donc on corrige et on redémarre
# nano /etc/fstab pour supprimer la ligne qui induit l'erreur
/dev/sdb1   /mnt/data   ext4    defaults    0   2
mount -a # vérification normalement pas d'erreur
reboot
```
- Conclusion générale :
    - /etc/fstab est lu à chaque démarrage pour monter les systèmes de fichiers
    - Une erreur dans le montage au redémarrage bascule le système en mode rescue
    - Toujours après la modification du fichier /etc/fstab exécuter mount -a pour tester que tout est ok
